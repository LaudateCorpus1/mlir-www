32 /// The Simplex class implements a version of the Simplex and Generalized Basis
33 /// Reduction algorithms, which can perform analysis of integer sets with affine
34 /// inequalities and equalities. A Simplex can be constructed
35 /// by specifying the dimensionality of the set. It supports adding affine
36 /// inequalities and equalities, and can perform emptiness checks, i.e., it can
37 /// find a solution to the set of constraints if one exists, or say that the
38 /// set is empty if no solution exists. Furthermore, it can find a subset of
39 /// these constraints that are redundant, i.e. a subset of constraints that
40 /// doesn't constrain the affine set further after adding the non-redundant
41 /// constraints. The LexSimplex class provides support for computing the
42 /// lexicographical minimum of an IntegerPolyhedron. Both these classes can be
43 /// constructed from an IntegerPolyhedron, and both inherit common
44 /// functionality from SimplexBase.
45 ///
46 /// The implementations of the Simplex and SimplexBase classes, other than the
47 /// functionality for obtaining an integer sample, are based on the paper
48 /// "Simplify: A Theorem Prover for Program Checking"
49 /// by D. Detlefs, G. Nelson, J. B. Saxe.
50 ///
51 /// We define variables, constraints, and unknowns. Consider the example of a
52 /// two-dimensional set defined by 1 + 2x + 3y >= 0 and 2x - 3y >= 0. Here,
53 /// x, y, are variables while 1 + 2x + 3y >= 0, 2x - 3y >= 0 are constraints.
54 /// Unknowns are either variables or constraints, i.e., x, y, 1 + 2x + 3y >= 0,
55 /// 2x - 3y >= 0 are all unknowns.
56 ///
57 /// The implementation involves a matrix called a tableau, which can be thought
58 /// of as a 2D matrix of rational numbers having number of rows equal to the
59 /// number of constraints and number of columns equal to one plus the number of
60 /// variables. In our implementation, instead of storing rational numbers, we
61 /// store a common denominator for each row, so it is in fact a matrix of
62 /// integers with number of rows equal to number of constraints and number of
63 /// columns equal to _two_ plus the number of variables. For example, instead of
64 /// storing a row of three rationals [1/2, 2/3, 3], we would store [6, 3, 4, 18]
65 /// since 3/6 = 1/2, 4/6 = 2/3, and 18/6 = 3.
66 ///
67 /// Every row and column except the first and second columns is associated with
68 /// an unknown and every unknown is associated with a row or column. An unknown
69 /// associated with a row or column is said to be in row or column orientation
70 /// respectively. As described above, the first column is the common
71 /// denominator. The second column represents the constant term, explained in
72 /// more detail below. These two are _fixed columns_; they always retain their
73 /// position as the first and second columns. Additionally, LexSimplex stores
74 /// a so-call big M parameter (explained below) in the third column, so
75 /// LexSimplex has three fixed columns.
76 ///
77 /// LexSimplex does not directly support variables which can be negative, so we
78 /// introduce the so-called big M parameter, an artificial variable that is
79 /// considered to have an arbitrarily large value. We then transform the
80 /// variables, say x, y, z, ... to M, M + x, M + y, M + z. Since M has been
81 /// added to these variables, they are now known to have non-negative values.
82 /// For more details, see the documentation for LexSimplex. The big M parameter
83 /// is not considered a real unknown and is not stored in the `var` data
84 /// structure; rather the tableau just has an extra fixed column for it just
85 /// like the constant term.
86 ///
87 /// The vectors var and con store information about the variables and
88 /// constraints respectively, namely, whether they are in row or column
89 /// position, which row or column they are associated with, and whether they
90 /// correspond to a variable or a constraint.
91 ///
92 /// An unknown is addressed by its index. If the index i is non-negative, then
93 /// the variable var[i] is being addressed. If the index i is negative, then
94 /// the constraint con[~i] is being addressed. Effectively this maps
95 /// 0 -> var[0], 1 -> var[1], -1 -> con[0], -2 -> con[1], etc. rowUnknown[r] and
96 /// colUnknown[c] are the indexes of the unknowns associated with row r and
97 /// column c, respectively.
98 ///
99 /// The unknowns in column position are together called the basis. Initially the
100 /// basis is the set of variables -- in our example above, the initial basis is
101 /// x, y.
102 ///
103 /// The unknowns in row position are represented in terms of the basis unknowns.
104 /// If the basis unknowns are u_1, u_2, ... u_m, and a row in the tableau is
105 /// d, c, a_1, a_2, ... a_m, this represents the unknown for that row as
106 /// (c + a_1*u_1 + a_2*u_2 + ... + a_m*u_m)/d. In our running example, if the
107 /// basis is the initial basis of x, y, then the constraint 1 + 2x + 3y >= 0
108 /// would be represented by the row [1, 1, 2, 3].
109 ///
110 /// The association of unknowns to rows and columns can be changed by a process
111 /// called pivoting, where a row unknown and a column unknown exchange places
112 /// and the remaining row variables' representation is changed accordingly
113 /// by eliminating the old column unknown in favour of the new column unknown.
114 /// If we had pivoted the column for x with the row for 2x - 3y >= 0,
115 /// the new row for x would be [2, 1, 3] since x = (1*(2x - 3y) + 3*y)/2.
116 /// See the documentation for the pivot member function for details.
117 ///
118 /// The association of unknowns to rows and columns is called the _tableau
119 /// configuration_. The _sample value_ of an unknown in a particular tableau
120 /// configuration is its value if all the column unknowns were set to zero.
121 /// Concretely, for unknowns in column position the sample value is zero; when
122 /// the big M parameter is not used, for unknowns in row position the sample
123 /// value is the constant term divided by the common denominator. When the big M
124 /// parameter is used, if d is the denominator, p is the big M coefficient, and
125 /// c is the constant term, then the sample value is (p*M + c)/d. Since M is
126 /// considered to be positive infinity, this is positive (negative) infinity
127 /// when p is positive or negative, and c/d when p is zero.
128 ///
129 /// The tableau configuration is called _consistent_ if the sample value of all
130 /// restricted unknowns is non-negative. Initially there are no constraints, and
131 /// the tableau is consistent. When a new constraint is added, its sample value
132 /// in the current tableau configuration may be negative. In that case, we try
133 /// to find a series of pivots to bring us to a consistent tableau
134 /// configuration, i.e. we try to make the new constraint's sample value
135 /// non-negative without making that of any other constraints negative. (See
136 /// findPivot and findPivotRow for details.) If this is not possible, then the
137 /// set of constraints is mutually contradictory and the tableau is marked
138 /// _empty_, which means the set of constraints has no solution.
139 ///
140 /// This SimplexBase class also supports taking snapshots of the current state
141 /// and rolling back to prior snapshots. This works by maintaining an undo log
142 /// of operations. Snapshots are just pointers to a particular location in the
143 /// log, and rolling back to a snapshot is done by reverting each log entry's
144 /// operation from the end until we reach the snapshot's location. SimplexBase
145 /// also supports taking a snapshot including the exact set of basis unknowns;
146 /// if this functionality is used, then on rolling back the exact basis will
147 /// also be restored. This is used by LexSimplex because its algorithm, unlike
148 /// Simplex, is sensitive to the exact basis used at a point.

355 /// Simplex class using the lexicographic pivot rule. Used for lexicographic
356 /// optimization. The implementation of this class is based on the paper
357 /// "Parametric Integer Programming" by Paul Feautrier.
358 ///
359 /// This does not directly support negative-valued variables, so it uses the big
360 /// M parameter trick to make all the variables non-negative. Basically we
361 /// introduce an artifical variable M that is considered to have a value of
362 /// +infinity and instead of the variables x, y, z, we internally use variables
363 /// M + x, M + y, M + z, which are now guaranteed to be non-negative. See the
364 /// documentation for Simplex for more details. The whole algorithm is performed
365 /// without having to fix a "big enough" value of the big M parameter; it is
366 /// just considered to be infinite throughout and it never appears in the final
367 /// outputs. We will deal with sample values throughout that may in general be
368 /// some linear expression involving M like pM + q or aM + b. We can compare
369 /// these with each other. They have a total order:
370 /// aM + b < pM + q iff a < p or (a == p and b < q).
371 /// In particular, aM + b < 0 iff a < 0 or (a == 0 and b < 0).
372 ///
373 /// Initially all the constraints to be added are added as rows, with no attempt
374 /// to keep the tableau consistent. Pivots are only performed when some query
375 /// is made, such as a call to getRationalLexMin. Care is taken to always
376 /// maintain a lexicopositive basis transform, explained below.
377 ///
378 /// Let the variables be x = (x_1, ... x_n). Let the basis unknowns at a
379 /// particular point be y = (y_1, ... y_n). We know that x = A*y + b for some
380 /// n x n matrix A and n x 1 column vector b. We want every column in A to be
381 /// lexicopositive, i.e., have at least one non-zero element, with the first
382 /// such element being positive. This property is preserved throughout the
383 /// operation of LexSimplex. Note that on construction, the basis transform A is
384 /// the indentity matrix and so every column is lexicopositive. Note that for
385 /// LexSimplex, for the tableau to be consistent we must have non-negative
386 /// sample values not only for the constraints but also for the variables.
387 /// So if the tableau is consistent then x >= 0 and y >= 0, by which we mean
388 /// every element in these vectors is non-negative. (note that this is a
389 /// different concept from lexicopositivity!)
390 ///
391 /// When we arrive at a basis such the basis transform is lexicopositive and the
392 /// tableau is consistent, the sample point is the lexiographically minimum
393 /// point in the polytope. We will show that A*y is zero or lexicopositive when
394 /// y >= 0. Adding a lexicopositive vector to b will make it lexicographically
395 /// bigger, so A*y + b is lexicographically bigger than b for any y >= 0 except
396 /// y = 0. This shows that no point lexicographically smaller than x = b can be
397 /// obtained. Since we already know that x = b is valid point in the space, this
398 /// shows that x = b is the lexicographic minimum.
399 ///
400 /// Proof that A*y is lexicopositive or zero when y > 0. Recall that every
401 /// column of A is lexicopositive. Begin by considering A_1, the first row of A.
402 /// If this row is all zeros, then (A*y)_1 = (A_1)*y = 0; proceed to the next
403 /// row. If we run out of rows, A*y is zero and we are done; otherwise, we
404 /// encounter some row A_i that has a non-zero element. Every column is
405 /// lexicopositive and so has some positive element before any negative elements
406 /// occur, so the element in this row for any column, if non-zero, must be
407 /// positive. Consider (A*y)_i = (A_i)*y. All the elements in both vectors are
408 /// non-negative, so if this is non-zero then it must be positive. Then the
409 /// first non-zero element of A*y is positive so A*y is lexicopositive.
410 ///
411 /// Otherwise, if (A_i)*y is zero, then for every column j that had a non-zero
412 /// element in A_i, y_j is zero. Thus these columns have no contribution to A*y
413 /// and we can completely ignore these columns of A. We now continue downwards,
414 /// looking for rows of A that have a non-zero element other than in the ignored
415 /// columns. If we find one, say A_k, once again these elements must be positive
416 /// since they are the first non-zero element in each of these columns, so if
417 /// (A_k)*y is not zero then we have that A*y is lexicopositive and if not we
418 /// ignore more columns; eventually if all these dot products become zero then
419 /// A*y is zero and we are done.
420 class LexSimplex : public SimplexBase {
421 public:
422  explicit LexSimplex(unsigned nVar)
423  : SimplexBase(nVar, /*mustUseBigM=*/true) {}
424  explicit LexSimplex(const IntegerPolyhedron &constraints)
425  : LexSimplex(constraints.getNumIds()) {
426  intersectIntegerPolyhedron(constraints);
427  }
428  ~LexSimplex() override = default;

433  ///
434  /// This just adds the inequality to the tableau and does not try to create a
435  /// consistent tableau configuration.
436  void addInequality(ArrayRef<int64_t> coeffs) final {

473 /// The Simplex class uses the Normal pivot rule and supports integer emptiness
474 /// checks as well as detecting redundancies.
475 ///
476 /// The Simplex class supports redundancy checking via detectRedundant and
477 /// isMarkedRedundant. A redundant constraint is one which is never violated as
478 /// long as the other constraints are not violated, i.e., removing a redundant
479 /// constraint does not change the set of solutions to the constraints. As a
480 /// heuristic, constraints that have been marked redundant can be ignored for
481 /// most operations. Therefore, these constraints are kept in rows 0 to
482 /// nRedundant - 1, where nRedundant is a member variable that tracks the number
483 /// of constraints that have been marked redundant.
484 ///
485 /// Finding an integer sample is done with the Generalized Basis Reduction
486 /// algorithm. See the documentation for findIntegerSample and reduceBasis.
487 class Simplex : public SimplexBase {
488 public:
489  enum class Direction { Up, Down };
490

501  /// c_n + c_0*x_0 + c_1*x_1 + ... + c_{n-1}*x_{n-1} >= 0.
502  ///
503  /// This also tries to restore the tableau configuration to a consistent
504  /// state and marks the Simplex empty if this is not possible.
505  void addInequality(ArrayRef<int64_t> coeffs) final;
506 
507  /// Compute the maximum or minimum value of the given row, depending on
508  /// direction. The specified row is never pivoted. On return, the row may
509  /// have a negative sample value if the direction is down.
510  ///
511  /// Returns a Fraction denoting the optimum, or a null value if no optimum
512  /// exists, i.e., if the expression is unbounded in this direction.
513  Optional<Fraction> computeRowOptimum(Direction direction, unsigned row);

517  ///
518  /// Returns a Fraction denoting the optimum, or a null value if no optimum

520  Optional<Fraction> computeOptimum(Direction direction,

523  /// Returns whether the perpendicular of the specified constraint is a
524  /// is a direction along which the polytope is bounded.
525  bool isBoundedAlongConstraint(unsigned constraintIndex);
526 
527  /// Returns whether the specified constraint has been marked as redundant.
528  /// Constraints are numbered from 0 starting at the first added inequality.
